It is not the number of students per se that is the issue, but that 2000 requests are all trying to execute at the same time and memory and database are getting overloaded. Say, one really simple and effective approach, I would take is by restricting the number of requests handled by the server, say with a queue or a concurrency limit.
So, rather than have 2000 requests run all at once, I would have only 50-100 requests run at once and queue the rest of the requests for execution later. This can be done using something like a semaphore inside FastAPI or a reverse proxy like Nginx that rate-limits incoming traffic.

Memory: things like request data, JWT decoding, database session, ORM objects are stored in each request of an app written in FastAPI. The average estimate for the amount of memory per active request is about 1-2 MB during peak usage. If all 2000 requests are executed concurrently, this could be as much as 2-4 GB of memory utilization, causing a 1 GB machine to crash and/or swapping heavily to become unresponsive.

If I set the concurrency to 50 requests, then the memory usage is approx. 50 \* 2 MB, meaning it's about 100 MB, which is fine. The rest of the requests will not be discarded rather they are stored in a queue and will be performed in order once the earlier requests are completed.

This also defends the database, as it can stall everything out if too many connections are made simultaneously, or exhaust the database connection. Basically, it's like if I had to let the traffic flow into the server instead of trying to handle everything at once and it crashes, I would be to manage the traffic so that the server could process the requests steadily over time. This helps to maintain a stable memory, prevents database overload, and keeps the system responsive during peak demand periods.
